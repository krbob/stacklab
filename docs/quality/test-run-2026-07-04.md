## Podsumowanie

Aplikacja jest w **bardzo dobrym stanie**. Cały baseline automatyczny zielony na
arm64, a testy manualne/eksploracyjne potwierdziły poprawność i wysoką jakość UX
w praktycznie wszystkich krytycznych ścieżkach — w tym najbardziej ryzykownych
przepływach uprzywilejowanych (docker daemon apply + **rollback**, naprawa
uprawnień, git commit/push). Nie znaleziono blokerów ani defektów łamiących
kryteria akceptacji. Znaleziono **5 drobnych/średnich usterek** (log-spam,
ergonomia logów/stats, kosmetyka danych) — żadna nie blokuje wydania.

## Znalezione usterki

Żadna nie jest blokerem. Priorytet do rozważenia produktowo.

### F-05 [bug / medium] — spam logów przy nieprawidłowym stacku
`failed to parse compose file` (dla stacka `invalid`) jest logowane na poziomie
**WARN przy każdym cyklu** discovery/stats: ~0,5–1/s w spoczynku, burst ~50/s
przy wielu otwartych widokach (62 wpisy w 2 min). Zalewa journald oraz wbudowany
podgląd logów na `/host`. **Rekomendacja:** rate-limit / log raz na zmianę stanu
/ obniżyć do DEBUG. Największy praktyczny mankament z listy.

### F-03 [ux / medium] — logi nie są sortowane chronologicznie między serwisami
W widoku Logs stacka wielo-serwisowego linie nowszego serwisa potrafią być
wyświetlone nad starszymi liniami innego serwisa (merge per-kontener bez
globalnego sortowania po znaczniku czasu). Utrudnia diagnostykę korelacji zdarzeń
między serwisami. **Rekomendacja:** scalać strumienie z sortowaniem po timestamp
(z rozsądnym oknem buforowania).

### F-04 [ux / low-med] — mianownik „Stack RAM" zawyża dostępną pamięć
Na `/stacks/:id/stats` karta STACK RAM pokazuje np. `5.1 MB / 7.6 GB`, podczas
gdy pojedyncze kontenery pokazują `/ 3.8 GB` (realna pamięć hosta). Total sumuje
limit hosta × liczba kontenerów, dając nierealny mianownik. **Rekomendacja:**
jako mianownik stacka użyć pamięci hosta (raz), nie sumy per-kontener.

### F-01 [bug / low] — zduplikowane porty w danych kontenera
`GET /api/stacks/{id}` → `.stack.containers[].ports` zawiera ten sam wpis
(np. 8081→80/tcp) dwukrotnie — z powodu osobnych bindów IPv4 (0.0.0.0) i IPv6
(::). Widok Overview używa portów z definicji serwisu i wyświetla poprawnie (raz),
więc obecnie **niewidoczne dla użytkownika**, ale warto zdeduplikować po
`published/target/protocol`, zanim jakiś widok zacznie renderować `container.ports`.

### F-02 [gap / known] — brak drift-baseline po udanym deploy
Po udanym `up` stack ma `config_state:"unknown"` i `last_deployed_at:null`.
Zgodne z backlogiem w `docs/roadmap.md` (persystencja znormalizowanych hashy
`compose.yaml`/`.env` po deployu nie jest jeszcze zaimplementowana) — **znana
luka, nie regresja**. Warto potwierdzić, że UI komunikuje „unknown" czytelnie.

## Wyniki wg obszarów planu (PASS)

**A. Auth/sesja** — login (A2), błędne hasło bez wycieku szczegółów (A3),
**lockout**: 5× 401 → 6× 429, poprawne hasło w oknie blokady też odrzucone (A4),
chroniony REST bez sesji → 401 (A10).

**B. Discovery/dashboard** — 6 stacków wykrytych z filesystemu bez seedu DB (B1);
`invalid` od razu z `config_state:invalid`; kafle z metrykami cpu/mem na żywo;
chipy All/Problems/Updates; empty states.

**C. Szczegóły stacka** — serwisy z compose + mapowanie kontener↔serwis (C1),
tryb image/build per-serwis (C2), porty/mounty (C3), healthcheck redisa wykryty i
zmapowany jako `healthy` (C4), `capabilities`/`available_actions` z backendu.

**D. Orphaned** — po usunięciu compose.yaml działającego stacka:
`display_state:orphaned`, Editor wyłączony (`can_edit_definition:false`),
logs/stats/terminal dostępne, `available_actions:["down"]` (D1–D3).

**E. Edytor** — ładowanie compose + `.env` z flagą `exists:false` gdy brak
(E1–E3); zapis niepoprawnego configu przechodzi, ale Save&Deploy zablokowany, a
banner walidacji czytelny (E4, E10); resolved-config z **draftu bez zapisu na
dysk** (E6, potwierdzone md5 pliku).

**F. Akcje/joby** — `up` realnych stacków → succeeded, realne kontenery (F1);
**locking**: druga akcja na zablokowanym stacku → 409 (F5); akcje na różnych
stackach niezależne (F6); dialog usuwania z poprawnym domyślnym zakresem (tylko
runtime), data-directory oznaczone jako danger (F12–F14).

**G. Progres jobów** — panel osiąga stan terminalny „Succeeded" z pełnym śladem
kroków; output pozostaje po zakończeniu; pasek postępu w headerze + timer w
sidebarze; restart (~10 s) poprawnie zakończony (G1).

**H/I/J. Logi/Stats/Terminal** — logi z timestampem + kolorem per serwis,
Pause/Clear, licznik linii; stats: totals + per-container CPU/RAM/Net z jawną notą
o zakresie historii; terminal: łączenie do kontenera, wybór shell, prompt `/ #`
(I/O potwierdzone też testami integracyjnymi PTY). Zob. F-03, F-04.

**K. Audyt** — wpisy per-stack i globalne z `job_id`, duration, result; fixtura
retencji: job z usuniętym detalem zwraca `retained:false` (K4); sesje terminala
audytowane jako `terminal_opened` bez treści komend (K6).

**L. Bezpieczeństwo** — path traversal w `stackId`/`path` → 404/400 (S1); WS bez
sesji → 401 (S2); obcy Origin na login → 403 (S3); terminal renderowany przez
XTerm; hasło jako hash Argon2id (kod + brak plaintextu).

**Workspace/Git** — naprawa uprawnień: helper (sudo) zmienił root:root 0600 →
właściciel `stacklab` (W4); git status/diff; **selektywny commit zawiera tylko
wskazany plik** (potwierdzone `git show --stat`); push bez upstream → jasny 409.

**Maintenance** — lista images (z flagą in_use), networks, prune-preview z
pozycjami i powodami (M3/M4); prune ze `scope` → job succeeded + audyt (M6).

**Image updates** — check porównuje digesty; po podmianie obrazu na starszy:
`updates.state:"available"`, `services_with_updates:1` dla właściwego stacka;
build-mode `web-build` poprawnie bez trackingu (N1/N2). Uwaga: check jest
asynchroniczny — rollup odświeża się po zakończeniu joba.

**Templates** — 2 szablony; utworzenie z „Web service" nadaje `nginx:stable`,

**Docker admin** — overview silnika + „no daemon.json" (DA1); validate z
`write_capability.supported:true` (DA2); **apply&restart** zapisał daemon.json,

**Rejestry** — login z błędnymi danymi → job `failed` z czytelnym błędem (R3).

**Notyfikacje** — realna dostawa webhooka na lokalny listener (JSON,

**Self-update** — poprawnie wyłączony dla instalacji tarball z komunikatem

**Host** — overview (Debian 13/arm64, docker/compose, CPU/RAM/dysk z paskami)

**UX przekrojowe** — responsywność desktop (sidebar) i mobile (bottom-nav,

## Uwaga metodologiczna

W trakcie sesji trzykrotnie błędnie podejrzewałem defekt z powodu użycia złej
nazwy pola JSON (m.in. `image_update_summary` zamiast faktycznego `updates`).
Po weryfikacji surowych odpowiedzi API wszystkie trzy okazały się poprawnym
działaniem. Wniosek na przyszłość: weryfikować schemat (dump raw JSON) przed
klasyfikacją czegokolwiek jako bug.
